# v0.2b — Results Grid + Schema Tree — Design

**Milestone:** Second slice of v0.2 (Query Experience), following `v0.2a` app shell/editor.

**Goal:** Make LithePG feel like a usable Postgres IDE without adding persistence yet: improve result navigation and add read-only schema exploration for the active connection.

## Scope

### In scope

- Read-only schema tree for the current database:
  - schemas
  - tables and views
  - columns with type/nullability/default when available
- A headless `SchemaIntrospector` service in `LithePGCore` backed by SQL against PostgreSQL catalogs / `information_schema`.
- `AppState` ownership of loaded schema metadata and refresh/error state.
- SwiftUI sidebar that renders the schema tree for the current connection.
- Results grid polish that remains pure SwiftUI/AppKit and headlessly testable where possible:
  - copy-friendly cell text
  - row count/truncation messaging
  - clearer empty/command/result states
- Tests for schema parsing/query mapping and AppState schema-refresh behavior.

### Out of scope

- Saved connections and query history (v0.3).
- Multi-tab workspace (v0.2c).
- Full keyboard shortcut catalog (v0.2c).
- Mutating schema actions (create/drop/alter).
- ERD/schema visualization (v1.1+).
- UI automation/clicking in CI; manual smoke remains dogfood-log based.

## Architecture

- `LithePGCore`
  - Add `SchemaItem` / `DatabaseSchema` value models.
  - Add `SchemaIntrospector` helper that uses an open `PostgresConnector` or a protocol-friendly query function.
  - Keep SQL read-only and scoped to current database.
- `LithePGApp`
  - Extend `AppState` with `schema`, `isLoadingSchema`, `schemaError`, `refreshSchema()`.
  - Trigger initial schema refresh after successful connect.
  - Add `SchemaSidebar` and embed it in `WorkspaceView` with `NavigationSplitView` or a conservative `HStack` fallback.

## Testing Strategy

- Unit-test schema model ordering/grouping without Postgres.
- Integration-test schema introspection behind `POSTGRES_TEST_URL` using temporary tables/views.
- AppState live smoke: connect, refresh schema, verify `public` schema and a known temporary/dogfood table when possible.
- Full `swift test` must pass with integration tests skipped by default.

## Exit Criteria

1. User can connect to Postgres and see a sidebar containing schemas, tables/views, and columns.
2. Schema refresh is explicit and safe; errors display without dropping the connection.
3. Results area remains usable with the existing 10,000-row cap and clearer status text.
4. Full `swift test` passes.
5. `POSTGRES_TEST_URL` live schema smoke passes against the Docker Postgres container.
6. Manual UI receipt is added to `docs/dogfood-log.md` when Omar checks the app locally.

## Open Questions

1. Should v0.2b display all schemas by default, or hide system schemas (`pg_catalog`, `information_schema`) unless toggled?
2. Should clicking/double-clicking a table insert a starter query, or should v0.2b keep the tree read-only only?
3. For results grid polish, is CSV copy/export important now, or should export wait until after schema sidebar lands?
