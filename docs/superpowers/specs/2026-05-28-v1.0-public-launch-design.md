# v1.0 — Public Launch — Design

**Roadmap source:** `docs/superpowers/specs/2026-04-18-roadmap-design.md` §6 v1.0.

## Goal

Turn the dogfood-ready, local-first LithePG app into a public macOS launch artifact that is installable, documented, governable, and safe to distribute without weakening the lean/private thesis proven through v0.5.

## Exit Criteria

- Codesigned + notarized macOS `.app` build.
- Distribution through GitHub Releases plus a Homebrew cask path.
- README includes current screenshots, quickstart, install instructions, and release evidence links.
- `CHANGELOG.md` covers v0.1 through v1.0.
- `SECURITY.md` has a vulnerability-reporting path with expected response timing.
- Light + dark theme support exists, with dark as the default.
- Governance and contribution artifacts exist: `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, DCO sign-off policy, issue templates, and PR template.
- Existing v0.5 gates stay green: `swift test`, dogfood check, local AI privacy posture, and binary budget.

## Current Baseline — 2026-05-28

- `v0.5` is tagged on `main` after local Ask-in-English, dogfood, test, privacy, and measurement receipts.
- The app already has a release packaging helper in `script/build_and_run.sh --package` that creates `dist/LithePG.app` and strips the copied executable.
- `.github/workflows/ci.yml` is currently `workflow_dispatch` only because GitHub Actions push/PR triggers were blocked by account billing/spending-limit settings.
- CI security scan job is defined, but public launch needs clear local fallbacks because CI may remain manual-only until account settings are resolved.
- Public notarization will require Omar-controlled Apple Developer credentials; implementation must keep those credentials outside the repo.

## Product Behavior

### Install and launch

1. User downloads `LithePG.app.zip` from a GitHub Release or installs through a Homebrew cask.
2. macOS Gatekeeper accepts the app as signed and notarized.
3. First launch opens the connection sheet with no network calls, no bundled AI model download, and no credential prompts beyond the user's Postgres connection details.
4. Documentation clearly explains local-first AI, separate model artifacts, and safe generated-SQL review.

### Appearance

- Default app appearance is dark.
- User can switch to light, dark, or system appearance from app settings/preferences or an obvious menu affordance.
- Tests cover appearance preference persistence and default behavior without UI automation where possible.

### Public contributor path

- README points contributors to governance docs.
- Issue templates collect bug/feature/security-adjacent information without asking users to paste secrets or database contents.
- PR template reminds contributors to run local tests, avoid secrets, and sign off commits if DCO is enforced.

## Architecture

```text
script/build_and_run.sh --package
  └─ dist/LithePG.app
      └─ codesign/notarytool wrapper script
          └─ release artifact: LithePG.app.zip

LithePGApp
  └─ AppearanceSettings / persisted preference
      └─ SwiftUI preferredColorScheme

Docs + GitHub metadata
  ├─ README.md screenshots/install/release links
  ├─ CHANGELOG.md
  ├─ SECURITY.md reporting policy
  ├─ CONTRIBUTING.md / CODE_OF_CONDUCT.md / DCO
  └─ .github/ISSUE_TEMPLATE + pull_request_template.md
```

## Signing and Notarization Strategy

- Add scripts that support signing/notarization, but do not store credentials in the repository.
- Read signing identity and notary profile/team configuration from environment variables or a documented local keychain profile.
- If credentials are absent, packaging scripts should fail clearly after producing an unsigned local artifact, not silently claim launch readiness.
- Verify notarization with `spctl --assess` and `stapler validate` when credentials are available.

## Distribution Strategy

- GitHub Release is the source of truth for versioned app zips, changelog, and checksums.
- Homebrew cask work should generate/update a cask template from the GitHub Release URL and SHA-256.
- If the Homebrew tap does not exist yet, document the manual/tap creation step and stop before making assumptions about external repo ownership.

## Privacy and Security Invariants

- Do not add telemetry, update pings, cloud inference, crash uploaders, or model downloads for v1.0.
- Do not log connection URLs, passwords, query results, schemas, prompts, or generated SQL by default.
- Any screenshots committed to docs must use seeded/dummy data only.
- Signing/notarization credentials must remain in local keychain/env only.

## Testing Strategy

- Unit tests for appearance preference defaults, persistence, and SwiftUI state wiring.
- Shell syntax tests for release scripts.
- Local package smoke: build/package app, verify app bundle structure, verify executable is stripped below budget.
- Full `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer swift test`.
- Full Swift and environment-gated database integration tests before the final v1.0 tag.
- Signing/notarization smoke is credential-gated and must report skipped vs passed explicitly.

## Non-Goals

- Sparkle auto-updater.
- App Store distribution.
- Advanced AI quality improvements beyond v0.5's local service boundary.
- Multi-table AI joins beyond two tables.
- Full theming customization beyond light/dark/system.
- Automated Homebrew tap creation without Omar's explicit target repo.

## Open Decisions / Expected Blockers

- Apple Developer signing identity and notarytool profile/team ID are required for real notarization.
- Homebrew cask target tap/repository needs Omar's choice if it is not the main repo.
- GitHub Actions remains account-setting blocked: push/PR triggers are disabled, and the latest manual `workflow_dispatch` recheck failed before any job steps or logs were produced. v1.0 must rely on local receipts until Omar clears the Actions setting.
